# Web ページ版 要件整理

2026-04-23 時点の要件整理です。前提は「個人利用」「独自サーバーなし」「重いローカル開発環境なし」です。

## ゴール

日本の落語公演情報を 1 つの Web ページに集約し、検索しやすくすることです。

対象に含めるもの:

- 寄席
- ホール落語イベント
- 独演会

## 利用前提

- 主な利用者は自分
- PC とスマホの両方で見たい
- ログイン機能は不要
- 維持コストは低くしたい

## 制約

### インフラ制約

- 独自サーバーは使わない
- 常時動くバックエンドは持たない
- DB 前提の構成は避ける

### 開発制約

- Node.js や Xcode のような重い前提は置かない
- できればビルド不要にする
- ソース作成と簡単な確認だけで進めたい

### データ制約

- 情報ソースはインターネット上の公開情報
- まずは公式サイト優先
- 公演本文の転載ではなく、構造化した事実情報とリンク中心

## 機能要件

### 必須機能

1. 公演一覧を表示できる
2. 日付順に並べられる
3. 公演詳細を見られる
4. 元の公式ページへ飛べる
5. カテゴリで絞り込める
6. 出演者名で検索できる
7. 会場名で絞り込める
8. 最終更新日時が分かる

### 推奨機能

1. 今日、今週、今週末の簡易フィルタ
2. エリア絞り込み
3. お気に入り保存
4. `localStorage` による閲覧状態保存

### 非機能要件

1. スマホで読みやすい
2. JavaScript が失敗しても致命的に壊れにくい
3. できるだけ単一ファイル構成に近く保つ
4. データ件数が数百件規模でも十分軽い

## 推奨構成

### フロント

- `index.html`
- `styles.css`
- `app.js`
- `events.json`

### 配信

- GitHub Pages を第一候補
- 次点で Netlify または Cloudflare Pages

### データ取得

MVP ではブラウザで直接スクレイピングしません。

理由:

- `CORS` で取得できないページがある
- サイトごとの HTML 差分が大きい
- エラー時にユーザー画面へ影響が直結する

そのため、まずは `events.json` を用意してページ側はそれを読むだけにします。

## 運用要件

### 最初の運用

- 手動で情報を確認
- `events.json` を更新
- 静的ホスティングへ反映

### 後で追加できるもの

- GitHub Actions による定期生成
- Google Apps Script による取得補助
- イベント重複排除の自動化
